# Playbook - Harness Engineering

> 正式交付场景下的多 Agent 协作开发流程。
> 来源：AI 开发好马配好鞍——Harness Engineering 实践

## 状态：实践中 🟢

## 适用场景

- 正式交付的项目
- 跨模块、跨工程改动
- 需要质量保障和审计追溯

## 不适用

- 十行以内的小改动
- 纯探索性质的原型
- 技术攻关和可行性验证

## 核心理念

**Humans steer. Agents execute.** 人掌舵，Agent 干活。

AI 搞砸时，解决办法不是"再跑一次试试"，而是退一步想"缺了什么护栏？怎么让 Agent 看得懂、绕不过？"

## 七个 Agent 角色

| Agent | 职责 | 模型选择原则 |
|-------|------|-------------|
| PM Orchestrator | 流程总控：协调阶段、拍板前进/回退 | 轻量模型，只做调度 |
| Requirement Analyst | 需求拆解、多义性分析、验收标准 | 文本分析，不碰代码 |
| Solution Architect | 模块划分、接口定义、风险预判 | 文档输出，不写实现 |
| Gate Reviewer | 开发前多维度审查 | 审查模板化 |
| Developer Agent | 按方案落地、编译自检 | **最强模型**，唯一写代码的角色 |
| Code Reviewer | 逻辑漏洞、需求遗漏、设计偏离 | 深度理解代码但不自己写 |
| QA Tester | 测试用例、缺陷分类、测试工程 | 偏文档 |

## 流程状态机

```
[需求分析] → [方案设计] → [闸门评估] → [开发实现] → [代码评审] → [测试验证] → [交付]
                              ↓                            ↓              ↓
                         回退到需求/方案               回退到开发       回退到开发
```

### 硬规矩

- 闸门有阻塞项 → 不许开工，必须回退
- 评审有必改项 → 不许进测试，必须回退
- 测试有阻塞缺陷 → 回退到开发
- 同一阶段连续回退 3 次 → 停下来跟用户汇报
- **发现问题的人不能自己修**

### 文档交接结构

```
docs/features/<功能名>/
├── 01_REQUIREMENT_ANALYSIS.md
├── 02_SOLUTION_DESIGN.md
├── 03_GATE_REVIEW.md
├── 04_DEVELOPMENT.md
├── 05_CODE_REVIEW.md
└── 06_TEST_REPORT.md
```

## 三层验证体系

| 层 | 是什么 | 干什么 |
|----|--------|--------|
| 流程 Rule | 告诉 AI "改完必须验证" | 保证流程不被跳过 |
| Skill | 告诉 AI "怎么验证" | 封装操作步骤 |
| Script | 执行实际检查 | 机械化检测，不靠 AI 记忆 |

**原则：能用机器查的就别靠 AI 记。**

## 流程重量匹配

| 任务类型 | 建议流程 |
|----------|----------|
| 大需求 / 跨工程 | 完整七阶段 |
| 中等需求 / 单工程 | 可省略闸门 |
| 小 Bug 修复 | 根因分析 → 开发 → 评审 → 测试 |
| 十行以内 | 直接改 + 跑验证 |
